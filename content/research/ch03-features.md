## 1. The workflow file format (frontmatter + Markdown body) — Concept 1

- A workflow is a single `.md` in `.github/workflows/`: a YAML **frontmatter** block between `---`
  markers, then a **natural-language Markdown body**. Verified shape (ch02 example): frontmatter with
  `on:`, `permissions:`, `engine:`, `network:`, `safe-outputs:`, then a `# Repo Assistant …` prose body.
- **Author rule:** the body *is* the program logic (interpreted at run time); the frontmatter is
  machine-facing config. Do not present the body as a mere "description field."

## 2. `gh aw compile` — the compile model — Concepts 2 & 4

Verified CLI behavior on v0.81.6:

- `gh aw compile <file.md>` → writes `<file>.lock.yml`, prints
  `✓ <file> (100.5 KB)` then `✓ Compiled 1 workflow(s): 0 error(s), 0 warning(s)`, exit 0.
- **Offline & deterministic:** no engine call, no GitHub API, no secrets needed. Works behind the SAML
  wall. This is *why* every book example is compile-verified.
- With **no argument**, compiles all workflows in `.github/workflows/`. `--dir <d>` / a dir argument
  compile a directory. (Also available: `gh aw compile --watch` to recompile on save — confirm flag
  name on the local build before demoing.)
- **Strict mode is the default** (lock metadata `"strict":true`). It *refuses* unsafe choices
  (top-level write perms, unpinned actions, wildcard egress, deprecated fields) and *supplies safe
  defaults* for omitted keys (no `engine:` → Copilot; no `permissions:` → read-only; no `network:` →
  curated allowlist).

## 3. What `.lock.yml` contains — Concepts 2, 3, 4 (VERIFIED, quote-accurate)

The generated lock (ch02 example) has, top to bottom:

**(a) Metadata header (line 1)** — a single `# gh-aw-metadata:` JSON comment:
```
# gh-aw-metadata: {"schema_version":"v4","frontmatter_hash":"a783ee73…","body_hash":"8c11b173…",
#   "compiler_version":"v0.81.6","strict":true,"agent_id":"copilot","engine_versions":{"copilot":"1.0.65"}}
```
→ carries the **frontmatter_hash** and **body_hash** (compile-determinism signal, Concept 3),
`compiler_version`, `strict`, and the resolved engine.

**(b) Dependency manifest (line 2)** — a `# gh-aw-manifest:` JSON comment listing `secrets`, every
`actions` entry with its pinned `sha` + human `version`, and `containers` with image `digest`s.

**(c) Human-readable "DO NOT EDIT" banner** (ASCII-art "Agentic Workflows") with the exact text:
`# This file was automatically generated by gh-aw (v0.81.6). DO NOT EDIT.` and
`# To update this file, edit the corresponding .md file and run: gh aw compile`.

**(d) Comment manifests** a reader can audit at a glance:
- `# Secrets used:` — e.g. `COPILOT_GITHUB_TOKEN`, `GH_AW_GITHUB_TOKEN`, `GITHUB_TOKEN`, …
- `# Custom actions used:` — **SHA-pinned**, e.g.
  `actions/checkout@9c091bb21b7c1c1d1991bb908d89e4e9dddfe3e0 # v7.0.0`,
  `actions/github-script@3a2844b7…895573f7108da1b3 # v9.0.0`.
- `# Container images used:` — digest-pinned, e.g. the gh-aw firewall (`squid`, `api-proxy`, `agent`)
  and MCP gateway images (`ghcr.io/github/gh-aw-mcpg:…@sha256:…`).

**(e) `name:` → `on:` → `permissions: {}` → `jobs:`.** Note the compiler emits **top-level
`permissions: {}`** (empty) and pushes scopes down into individual jobs — least privilege made concrete.
The `on:` block expands `workflow_dispatch:` with generated `inputs:` (e.g. `aw_context`).

**(f) The jobs graph — the determinism boundary made concrete (Concept 3).** Verified job names:
| Job | Role | Deterministic? |
|---|---|---|
| `pre_activation` | pre-flight gating | ✅ fixed |
| `activation` | resolves trigger context | ✅ fixed |
| **`agent`** | **the engine runs the prompt (read-only)** | ❌ **the one model-driven step** |
| `detection` | threat / safe-output detection | ✅ fixed |
| `safe_outputs` | validates + applies requested writes (scoped) | ✅ fixed |
| `conclusion` | finalize / report | ✅ fixed |
→ The teaching point: **exactly one job (`agent`) is non-deterministic**; everything around it —
including the write path (`safe_outputs`) — is fixed, SHA-pinned, and auditable.

## 4. `gh aw status` / `gh aw list` — observe compiled state — Concept 5

- `gh aw status` — reports each workflow's state (enabled/disabled, schedules, labels); with a ref, the
  latest run status (uses the API).
- `gh aw list` — fast, no-API sibling: name / engine / **compilation status**.
- (Author caution: keep these light; ch12 owns observability/`gh aw logs`. Confirm exact columns on the
  local build before showing sample output — do not fabricate a table.)

## 5. The authoring loop + fast/slow path — Concept 5

- Loop: **write → compile → (status/list) → run → iterate.**
- **Fast path:** editing the **Markdown body** takes effect on the *next run* — no recompile
  (the lock loads the body at runtime).
- **Slow path:** changing **frontmatter** requires `gh aw compile` again (it is security-sensitive
  configuration baked into the lock). Rule of thumb: *edit body freely; recompile after any frontmatter
  change.*
- `gh aw run` is a **live** step (needs the engine secret, spends credits) — **not** compile-verified,
  and requires a `workflow_dispatch` trigger (ch02 already established this).
